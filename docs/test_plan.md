## 1. Введение (Introduction)
### 1.1 Цель документа

Этот документ описывает стратегию, объем, подход и ресурсы для тестирования веб-приложения "SauceDemo". План тестирования предназначен для координации работы команды и обеспечения качества продукта.

### 1.2 Обзор проекта

"SauceDemo" — это интернет-магазин, который позволяет пользователям просматривать товары, добавлять их в корзину и оформлять заказы. Проект находится на этапе разработки, и тестирование необходимо для обеспечения корректной работы всех функций.

## 2. Объем тестирования (Scope)
### 2.1 Компоненты для тестирования

- Авторизация пользователя (Login)
- Каталог товаров (Inventory)
- Работа с корзиной (Cart)
- Процесс оформления заказа (Checkout)
    

### 2.2 Компоненты, не входящие в объем тестирования

- Интеграция с внешними платежными системами (тестируется отдельно).  
- Мобильная версия сайта (тестируется в следующей фазе).

## 3. Подход к тестированию (Test Approach)
### 3.1 Типы тестирования

- **Дымовое тестирование (Smoke Testing)**: Быстрая проверка работоспособности продукт после сборки.  
- **Функциональное тестирование**: Проверка работы всех функций интернет-магазина.
- **Регрессионное тестирование**: Проверка, что новые изменения не сломали существующую функциональность.
- **Тестирование пользовательского интерфейса (UI)**: Проверка корректности отображения элементов интерфейса.
- **Тестирование удобства использования (UX)**: Оценка удобства навигации и взаимодействия с сайтом.
    

### 3.2 Методы тестирования

- Ручное тестирование.
- Автоматизированное тестирование (для регрессионных тестов).

## 4. Критерии начала и завершения тестирования (Entry and Exit Criteria)
### 4.1 Критерии начала тестирования

- Все основные функции реализованы и доступны для тестирования.
- Тестовая среда настроена и готова к использованию.
- Тестовые данные подготовлены.
    
### 4.2 Критерии завершения тестирования

- Все запланированные тесты выполнены.
- Нет критических дефектов (blocker/critical).
- Сформирован отчёт тестирования

## 6. Риски (Risks)
### 6.1 Потенциальные риски

- Недостаток тестовых данных.
- Задержки в поставке новых версий для тестирования.
- Недостаточная производительность тестовой среды.
    
### 6.2 Способы минимизации рисков

- Подготовка тестовых данных заранее.
- Регулярная синхронизация с командой разработки.
- Мониторинг и оптимизация тестовой среды.

## 7. Отчетность (Reporting)
### 7.1 Виды отчетов

- Ежедневные отчеты о ходе тестирования.
- Генерация отчётов через Allure Report/ Включает: результаты тестов, шаги выполнения, скриншоты при падениях
- Итоговый отчет по завершении тестирования.
    

### 7.2 Получатели отчетов

- Руководитель проекта.
- Заинтересованные стороны.

## 8. Приложения (Appendices)

- **Тест-кейс**: Документ, описывающий сценарий тестирования.
- **Чек-лист**: Упрощенный список проверок без подробных шагов.
